# MQTT 5.0 vs MQTT 3.1.1 Feature Matrix

This document lists every feature in this library and indicates whether
it is available when a client connects over MQTT 5.0 (`ProtocolV5`) or
MQTT 3.1.1 (`ProtocolV311`).

The server accepts only MQTT 5.0 by default. Use
`WithAcceptProtocolVersions(ProtocolV5, ProtocolV311)` to also accept
v3.1.1 clients. Both protocol versions share the same topic namespace;
v5-only packet fields and properties are automatically stripped when a
message is delivered to a v3.1.1 client.

## Control Packets

| Packet | v5 | v3.1.1 | Notes |
|--------|----|--------|-------|
| CONNECT | Yes | Yes | Shared `ConnectPacket` struct. v3.1.1 CONNECT has no properties section and no Will properties. |
| CONNACK | Yes | Yes | v3.1.1 CONNACK is always 4 bytes: flags + return code, no properties. |
| PUBLISH | Yes | Yes | v3.1.1 PUBLISH has no properties section. |
| PUBACK | Yes | Yes | v3.1.1 PUBACK is always 4 bytes. No reason code, no properties. |
| PUBREC | Yes | Yes | v3.1.1 PUBREC is always 4 bytes. No reason code, no properties. |
| PUBREL | Yes | Yes | v3.1.1 PUBREL is always 4 bytes. No reason code, no properties. |
| PUBCOMP | Yes | Yes | v3.1.1 PUBCOMP is always 4 bytes. No reason code, no properties. |
| SUBSCRIBE | Yes | Yes | v3.1.1 SUBSCRIBE has no properties. Reserved bits of the options byte must be zero. |
| SUBACK | Yes | Yes | v3.1.1 SUBACK return codes: `0x00`, `0x01`, `0x02`, `0x80`. No properties. |
| UNSUBSCRIBE | Yes | Yes | v3.1.1 UNSUBSCRIBE has no properties. |
| UNSUBACK | Yes | Yes | v3.1.1 UNSUBACK is always 4 bytes: no per-topic reason codes, no properties. |
| PINGREQ | Yes | Yes | 2-byte fixed packet in both versions. |
| PINGRESP | Yes | Yes | 2-byte fixed packet in both versions. |
| DISCONNECT (client to server) | Yes | Yes | v3.1.1 DISCONNECT is always 2 bytes (no reason code, no properties). |
| DISCONNECT (server to client) | Yes | No | MQTT 3.1.1 does not define a server-to-client DISCONNECT. The server closes v3.1.1 connections without writing one. |
| AUTH | Yes | No | Enhanced authentication (SCRAM, SASL) requires v5. |

## Cross-Version Delivery

When a v5 publisher sends a message to a v3.1.1 subscriber, the broker
delivers the topic name, payload, QoS, and retain flag but drops all
v5-specific fields:

- Payload Format Indicator
- Message Expiry Interval
- Topic Alias
- Response Topic
- Correlation Data
- Subscription Identifier
- Content Type
- User Properties

The reverse direction (v3.1.1 publisher to v5 subscriber) produces a v5
PUBLISH with an empty properties section — no data loss, just no extra
metadata to carry.
